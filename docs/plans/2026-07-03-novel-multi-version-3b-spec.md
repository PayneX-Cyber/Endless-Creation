# 小说创作 3b：章节多版本草稿 切片规格

日期：2026-07-03
上位文档：`docs/plans/2026-06-30-novel-creation-migration-plan.md`、`docs/plans/2026-07-02-novel-generation-chain-3a-spec.md`、`docs/plans/2026-07-02-novel-chapter-workbench-3a3-spec.md`

## 总体结论

3b 只做一件事：**把 3a.3 章节工作台的「单草稿预览」升级为「多版本候选草稿：可再生成、可比较、选定后写入正文」**。它不是版本管理系统，不是评审系统，不是批量生成器。

一句话目标：

```text
按顺序生成章节正文 → 可再生成多个候选版本 → 比较后选定一版写入正文 → 写入后版本保留，可回看与再次写回
```

这是 6-30 总纲第三阶段的后半刀（3a 规格拆分时明确后置的 `versions` / `selectedVersionId`）。3a 已跑通单版本生成链并验收，3b 在其上做增量，**不重构工作台，不动生成链**。

范围锁定为：**versions 字段落库 + 生成端多版本候选 + 选定写入 + 已完成章节版本回看/写回**。其它都不碰。

## 一、数据口径

`src/types/novel.ts`：

```ts
interface ChapterVersion {
  id: string;
  content: string;
  createdAt: string;
}

interface Chapter {
  // 既有字段不变
  versions?: ChapterVersion[];      // 新增：候选草稿版本（追加式）
  selectedVersionId?: string;       // 新增：最近一次写入正文的版本
}

interface Novel {
  version: 3;                       // 2 → 3
}
```

定死的口径：

- **`chapter.content` 仍是唯一正文事实源。** 字数、完成度、状态推导、编辑器、自动保存链路全部不变。「选定版本」= 把该版本 content 复制写入 `chapter.content`，仅此而已。
- **版本是草稿快照，不是同步副本。** 写入后用户手动编辑正文，版本内容不跟随变化；`selectedVersionId` 只表示"正文最近来自哪个版本"（溯源标记），正文被手动改后该标记继续保留。
- **手动编辑不产生版本。** 只有 AI 生成产生版本；不做自动快照。
- **版本落库是纯追加**：只写 `chapter.versions` / `selectedVersionId`，不覆盖任何既有字段。因其非破坏性，**每个版本生成完成即落库**（草稿重启不丢）；写入正文仍需显式确认，防覆盖口径沿用 3a.3（含 7d8c6eb 的持久层核对）。
- **每章版本上限 5**：达到上限后「再生成一版」置灰并提示；不做自动淘汰、不做版本删除（删除管理后置）。
- 不新增：版本 label、评分、prompt 记录、parentId、diff 数据、Bible、summary 链。

### v2→v3 sanitize 规则（沿用 3a 迁移模式）

```text
versions：非数组归一 undefined；数组内非法项过滤；content 非字符串的项丢弃
selectedVersionId：非字符串归一 undefined；指向不存在版本时保留字段但 UI 忽略
version：运行态归一 3
其余字段：保持 3a 规则
```

- **读取不写回**：只打开不编辑，磁盘 `novel.json` 不得被自动改成 v3。
- Web fallback 校验不得因 `version: 3` 或新字段过滤数据。

## 二、用户流程

生成端（3a.3 工作台右侧，不新增页面、不新增路由）：

```text
按顺序生成 → 草稿完成即落库为版本 1，进入版本预览
→ 可「再生成一版」（串行，同一 prompt，生成中禁止重复提交、可取消）
→ 版本切换条比较各版本（版本 1 / 版本 2 / …）
→ 「确认写入」= 将当前展示的版本写入正文（走防覆盖确认）
→ 状态变已完成，完成度更新，全部版本保留
→ 「关闭预览」仅收起面板，已落库版本不丢
```

回看端（已完成章节）：

```text
正文编辑器 meta 行显示「历史版本」入口（该章有版本才显示，无版本不出假入口）
→ 查看版本列表（时间、字数、当前选定标记）
→ 可将任一版本写回正文：走同一防覆盖确认；取消则正文不动
```

顺序锁、大纲生成、失败重试、错误脱敏等 3a.3 行为全部不变；**生成后续大纲不产生版本**（版本只属于正文）。

## 三、AI 行为

**零新增 prompt、零新增 IPC、零新增 Provider。**

- 「再生成一版」= 复用 `buildChapterFromOutlinePrompt` + `generateText` 原样再调一次（参数不变）。
- 不做并发生成、不做"一键生成 N 版"、不做版本间自动比较/评分。
- 失败处理沿用：显示重试、错误脱敏、已落库版本与 content/outline 不丢。

## 四、视觉要求

延续 3a.3 工作台风格：版本切换用轻量 pill 条，不做左右并排 diff 视图，不做表格式版本管理面板。「历史版本」入口保持 meta 行轻量链接级别，不加重编辑器。

## 五、后置清单（不进 3b）

- 版本 diff 对比视图、版本评分 / 评审（阶段四）
- 版本删除 / 重命名 / 导出、手动编辑自动快照
- 并发 / 批量生成、任务模型路由
- promptHistory、per-version 生成参数记录
- 世界观 / 角色卡 / Bible、一致性审校、RAG（阶段四之后）

**不做假入口：以上一律不展示，不置灰占位。**

## 遗留登记（非 3b 范围，需另行排期）

- **删除章节无入口**：3a.3 接受的债（旧工作台删除章节未搬迁且无其它归属），建议作为独立小切片处理。
- **生产加载白屏**：`vite.config.ts` 无 `base` 配置，`dist/index.html` 绝对 `/assets` 路径在 Electron `loadFile` 模式下 404。日常与 QA 均走 dev server 未暴露，打包发布前必须修（应用级问题，与小说模块无关）。

## 六、验收标准

生成端：

1. 按顺序生成完成后，草稿即版本 1 并已落库，预览区出现版本切换条。
2. 「再生成一版」串行产出新版本，生成中禁止重复提交、可取消；新版本生成后自动展示。
3. 版本达到 5 个后「再生成一版」置灰并提示上限。
4. 「确认写入」写入当前展示版本；防覆盖口径不回归——正文（内存或持久层）与生成快照不一致时弹确认，取消不写入。
5. 写入后状态已完成、完成度更新，`versions` 与 `selectedVersionId` 落库；重启后版本仍在。
6. 关闭预览不丢已落库版本。

回看端：

7. 已完成且有版本的章节显示「历史版本」入口；无版本的章节（含 3a 存量已完成章节）不显示。
8. 版本列表可查看各版本并将任一版本写回正文，写回走防覆盖确认；取消不改动正文。
9. 手动编辑正文不产生新版本、不改动已有版本，`selectedVersionId` 保留。

迁移（QA 重点）：

10. v2 旧小说打开正常；**只打开不编辑，磁盘不落 v3**；编辑保存后 `version` 变 3。
11. 缺失 / 非法 `versions`、`selectedVersionId` 不崩：按 sanitize 规则归一，UI 显示无版本态。

边界：

12. 零新增 prompt / IPC / Provider / schema 之外字段；不做 diff、评分、批量、版本删除、自动快照。
13. 不破坏 3a 全链路（灵感模式、项目查看、章节工作台顺序生成与大纲补齐）、小说 CRUD、生图与资产模块。

## 建议实施顺序

1. 架构确认：`ChapterVersion` 字段、v2→v3 sanitize 与「读取不写回」口径。
2. 前端：先做生成端版本落库 + 切换条（离 3a.3 草稿预览最近），再补回看端入口。
3. QA：复验 3b 验收 13 条 + v2 迁移，重点回归防覆盖与 3a 链路，不复验后续阶段。
