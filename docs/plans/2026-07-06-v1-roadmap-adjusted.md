# Endless Creation V1.0 执行路线图调整稿

日期：2026-07-06
定位：把 V1.0 愿景收敛为可狗粮、可内测、可逐步交付的执行路线。

## 一句话原则

先稳定小说/剧本创作闭环，再补商业化基建；先记录成本和导出成果，再做账户、OSS、支付和高级动效。

## V1.0 北极星

Endless Creation 是面向编剧、导演、小说作者的 AI 创作工作台。V1.0 不追求一次性完成“虚拟剧组”的全部想象，先证明一条主链路：

导入素材 -> 提取设定 -> 辅助创作 -> 生成多模态资产 -> 导出可交付成果。

## 阶段门

### Phase 1：MVP 狗粮闭环

目标：真实小说/剧本能跑完整主流程，不被阻断性 Bug 卡住。

验收：

- 导入一篇真实长文本后，项目可保存、关闭、重开，核心数据不丢。
- 能提取或维护角色、场景、伏笔、章节等基础设定。
- 能完成至少一次文本生成，并把结果写回当前项目。
- 能完成至少一次图像生成，并能在项目里找到产物。
- 失败时有明确错误提示，不白屏、不静默失败。
- `npm.cmd run build` 通过。

暂不做：

- 支付、订阅、复杂账户体系。
- OSS/S3 替换本地存储。
- 全量 Framer Motion 动效。
- 多 Agent 大重构。

### Phase 2：Beta 必需基建（已完成核心项）

目标：产品可以给 3-5 位种子用户试用，同时能控制成本、交付成果。

执行顺序：

1. Token 成本追踪最小闭环。
2. Word 分镜本导出。
3. ZIP 离线资源包导出。
4. 最小账户/JWT 闭环。
5. OSS/S3 存储替换。

2026-07-09 状态：

- 已完成：Token 成本追踪最小闭环、项目级成本看板、Markdown 导出、Word 分镜本导出、ZIP 离线包导出。
- 未进入：账户/JWT、OSS/S3、支付/套餐。它们仍属于真实 Beta 反馈后的规模化能力，不阻塞本地种子用户狗粮。

验收：

- 每次 AI 调用记录项目 ID、provider、model、input tokens、output tokens、估算成本、创建时间。
- 有项目级成本看板，可按项目查看总消耗和调用明细。
- 能导出 Word 分镜本，内容包含章节/镜头文本、角色、场景、图片引用。
- 能导出 ZIP 离线包，HTML 可本地打开，资源路径不丢。
- 登录态可维持，退出/过期后不会误用用户数据。
- 云存储接入前，本地 outputs 仍保持可用。

暂不做：

- 充值、套餐、复杂计费规则。
- 多租户企业权限。
- CDN、转码队列、对象生命周期管理。

### Phase 3：演示体验升级（已完成）

目标：把核心链路做出 Aha Moment，适合路演和正式展示。

执行顺序：

1. SSE 流式生成接口。
2. 前端打字机实时渲染。
3. 关键等待状态、骨架屏、错误恢复优化。
4. 人物关系图谱的 AI 推演。
5. 局部动效打磨。

验收：

- 文本生成能边生成边显示，取消后不会继续写入旧结果。
- 前端状态不会因为切换章节/项目串线。
- 图谱推演必须基于当前项目设定，不凭空改写角色关系。
- 动效只服务关键状态变化，不影响密集工作台效率。

暂不做：

- 全站动效重写。
- 重型 Agent 编排平台。
- 复杂移动端适配。

2026-07-09 状态：

- 已完成：SSE 流式生成、打字机实时渲染、取消/切章防串线、骨架屏、关键错误恢复、人物关系图谱 V0、项目详情/章节切换局部动效。
- 图谱 V0 使用当前项目蓝图与正文推演，按 `novelId` 保存到本地 `localStorage`；不是临时 session 态。

### Phase 4：规模化与商业化（后置）

目标：在核心创作、导出、资产、数据持久化等产品功能完善后，再验证付费路径和跨端场景。

2026-07-09 决策：Phase 4 支付/套餐/账户商业化整体后置；在所有核心功能完善并经过真实种子用户反馈前，不启动支付系统开发。

候选任务：

- 微信/支付宝/Stripe 支付。
- 套餐、额度、账单。
- 移动端查看生成进度。
- 小程序或响应式移动版。
- 更完整的对象存储、清理、归档策略。

进入条件：

- [ ] 情感曲线闭环：按章节 AI 分析情绪、按 novelId 持久化、支持重新分析并展示真实曲线；完成前不得进入 Phase 4。
- Phase 2 Beta 有真实用户反馈。
- 单用户成本和留存数据可观测。
- 主流程没有阻断性 Bug。
- 核心创作、导出、资产管理、本地数据恢复等产品功能已完善到可长期试用，不再用支付系统抢占产品完善优先级。

## 下一阶段：V1 Beta 收口包

目标：在进入 Phase 4 前，把“种子用户能独立完成一个项目”的缺口一次性复核清楚，只修阻断或明显体验 Bug，不新增支付/套餐/OSS。

范围：

- 导出闭环复核：Markdown、Word、ZIP 三条入口、结果、失败提示。
- 成本闭环复核：真实 AI 调用后单本小说看板稳定非零，按小说聚合且不串账。
- 本地数据风险检查：小说正文、人物图谱、AI usage 刷新/重启后不丢。
- 狗粮二轮 QA：创建 -> 生成 -> 图谱 -> 导出完整跑一遍。
- 路线图同步：持续删除过时“小刀派单”口径，按整包推进。

研发规则：

- 默认整包交付，不再把同一阶段拆成 design/protocol/UI/QA 小刀。
- Design/plan is not a standalone delivery unless PO explicitly asks for design-only work; once direction is accepted, implement and validate the full package.
- 只在互斥或高风险范围选择时请 PO 拍板；普通收口项直接执行。
- 每个收口包仍保持提交清晰、验证完整。

## 风险排序

1. 成本可见性已完成最小闭环；Beta 收口继续验证真实调用数据是否稳定。
2. AI 输出污染项目数据：所有结构化 AI 写入都必须先候选、再用户确认。
3. 多 Agent 失控：V1.0 只把 Agent 当后台分工，不把复杂编排暴露给用户。
4. 本地存储膨胀：Beta 前评估 outputs 体积，再决定 OSS 切换时机。
5. UI 过度打磨拖慢主链路：动效放 Phase 3，且只做关键状态。

## V1.0 成功标准

- PM 能用真实素材完成一次从导入到导出的完整狗粮测试。
- 种子用户能独立完成至少一个项目，不需要研发在旁边救火。
- 每次 AI 调用成本可追踪。
- 所有生成结果可导出。
- 主流程构建稳定，阻断 Bug 清零。
