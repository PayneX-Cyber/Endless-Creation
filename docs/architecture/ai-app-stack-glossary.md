# AI 应用技术栈备忘

日期：2026-07-06

这份文档只解释术语和适用时机，不代表当前项目已经决定引入这些技术。

## 前端层

| 技术 | 是什么 | 用途 |
| --- | --- | --- |
| React | 前端 UI 框架 | 写页面、按钮、弹窗、工作台等交互界面。 |
| Vite | 前端开发与构建工具 | 启动开发服务器、打包前端代码。 |
| Tailwind CSS | CSS 工具库 | 用 class 快速写布局、间距、颜色等样式。 |
| Framer Motion | React 动画库 | 做弹窗、卡片、页面切换等动效。 |
| Zustand | React 状态管理库 | 让多个组件共享运行时状态，如当前项目、设置、任务状态。 |

当前项目已经使用 React + Vite。Tailwind、Framer Motion、Zustand 不是必须项，只有现有 CSS / props / 局部 state 明显不够时再引入。

## 后端与数据层

| 技术 | 是什么 | 用途 |
| --- | --- | --- |
| FastAPI | Python 后端框架 | 提供 HTTP API，如生成、检索、项目数据接口。 |
| SQLite | 轻量本地数据库 | 用单文件数据库存结构化数据。 |
| SQLAlchemy | Python ORM | 用 Python 类和查询接口读写数据库。 |

常见分层：

```text
API -> Service -> Repo -> DB
```

- API：接收请求，做参数校验，返回响应。
- Service：处理业务规则，例如生成流程、权限、任务编排。
- Repo：封装数据读写，不让业务层直接碰 SQL。
- DB：真正的数据存储。

当前项目仍以 Electron 本地文件和现有 bridge 为主。只有当数据关系复杂、需要跨端访问、长任务服务化或多人协作时，才考虑 FastAPI + SQLite/SQLAlchemy。

## AI 能力层

| 技术 | 是什么 | 用途 |
| --- | --- | --- |
| RAG | 检索增强生成 | 先从资料库找相关内容，再交给 AI 生成，减少瞎编。 |
| FAISS | 向量检索库 | 快速查找语义相近的文本片段，常用于 RAG。 |
| Multi-Agent | 多智能体协作 | 让多个 AI 角色分工，如规划、写作、审稿、校对。 |

RAG/FAISS 适合资料量大到不能一次塞进 prompt 时再上。Multi-Agent 适合单个 AI 流程已经跑通、但确实需要多角色分工时再上。

## 实时通信层

| 技术 | 是什么 | 用途 |
| --- | --- | --- |
| SSE | 服务端单向流式推送 | 后端持续把生成文本推给前端，适合 AI 打字机式输出。 |
| WebSocket | 双向长连接 | 前端和后端持续互发消息，适合协作、实时状态、多用户互动。 |

AI 文本流式输出优先考虑 SSE。只有需要前后端双向实时通信时，再考虑 WebSocket。

## 引入顺序建议

1. 继续打磨当前 React/Electron 本地能力。
2. 数据结构明显复杂后，再评估 SQLite/SQLAlchemy。
3. 需要服务化、跨端访问或长任务队列时，再引入 FastAPI。
4. 小说资料太多、prompt 放不下时，再引入 RAG/FAISS。
5. AI 输出需要流式显示时，优先 SSE。
6. 需要双向实时协作时，再用 WebSocket。
7. Multi-Agent 最后引入，等单智能体流程已经稳定后再拆角色。

## 当前原则

不要因为技术栈听起来完整就一次性上齐。每引入一层，都要回答一个问题：现有 Electron + React + 本地保存链是否已经不够用了？
